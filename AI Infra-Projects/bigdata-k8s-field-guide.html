Part 1

Docker & Kubernetes Foundations

01

Containers & Docker

Infra world Chapter 1

Pure revision for you — the book opens here so data engineers have the same footing you started from. Skim it; the only thing worth noting is how every data tool later ships as a container image.

Container

A lightweight, isolated package holding an app plus everything it needs to run — the unit every tool in this book is shipped and deployed as.

Docker

The tooling that builds and runs containers on a single machine — your local sandbox before anything reaches Kubernetes.

Docker Image

The immutable, read-only blueprint a container is launched from — built once, run anywhere identically.

Dockerfile

A plain-text recipe of steps that builds an image — you'll write these for the custom Python ingestion jobs in the book.

Image Layer

Each Dockerfile instruction adds a cached filesystem layer; shared layers are reused, keeping images small and builds fast.

Container Registry

The store you push images to and pull them from (Docker Hub, Amazon ECR) — how images travel from your laptop to the cluster.

Immutable Infrastructure

The idea that you replace containers rather than patch them in place — the property that makes data pipelines reproducible.

Docker Daemon

The background service that actually builds, runs, and manages containers, driven through Docker's REST API/CLI.

docker-compose

Defines and runs a multi-container app from one YAML file — the book uses it to spin up local Kafka + Zookeeper clusters before K8s.

Container Runtime

The lower-level engine (containerd, CRI-O) that runs containers; Kubernetes calls it on each node to start your pods.

Job vs Service

Two workload shapes: a job runs to completion then stops (a Spark batch), a service stays up serving requests (an API). The distinction drives everything downstream.

02

Kubernetes Core

Infra world Chapters 2–3

Entirely your territory. Included for completeness and to fix vocabulary the data chapters lean on — StatefulSets, PersistentVolumes, and the operator foundation matter most once data workloads arrive.

Kubernetes

K8s

The container orchestrator that schedules, scales, heals, and connects workloads across a fleet of machines — the platform this whole book builds on.

Cluster

A set of machines Kubernetes manages as one pool — a control plane plus worker nodes.

Control Plane

The brain of the cluster — the components that decide what runs where and keep reality matching your declared intent.

API Server

The front door to the cluster; every kubectl command and operator talks to it. It's what CRDs extend.

etcd

The distributed key-value store holding all cluster state — the single source of truth, backed up in any serious DR plan.

Scheduler

Decides which node each new pod lands on based on resources and constraints — the same logic that places Spark executors.

Controller Manager

Runs the control loops that drive current state toward desired state — the pattern operators borrow to manage data tools.

Worker Node

A machine that actually runs your pods; data workloads are hungry, so node sizing and autoscaling matter here.

kubelet

The agent on each node that starts pods and reports their health back to the control plane.

kube-proxy

Handles pod-to-pod and service networking on each node so components can find each other.

Pod

The smallest deployable unit — one or more containers sharing a network address. Every Spark executor and Kafka broker runs as a pod.

Deployment

Manages a stateless, replicated, self-healing set of pods — right for APIs and Trino workers, wrong for stateful brokers.

Service

A stable network name and load-balancer in front of a changing set of pods — how components address each other.

Job

A pod that runs to completion for batch work — conceptually what a Spark batch run is.

StatefulSet

Like a Deployment but with stable identities and storage per pod — the correct primitive for stateful data services like Kafka.

↳ FOR YOU This is the K8s object that makes running databases and brokers on the cluster viable — worth a close look before §7.

PersistentVolume

PV

Cluster storage that outlives a pod — where Kafka logs and Spark shuffle data actually live.

PersistentVolumeClaim

PVC

A pod's request for a chunk of persistent storage; the JBOD Kafka setup defines several of these per broker.

ConfigMap

Non-secret configuration injected into pods, decoupled from the image — you already template these for tuning.

Secret

Like a ConfigMap but for sensitive values (DB passwords, API keys) the data connectors need.

Namespace

A logical partition of the cluster; the book isolates spark-operator, kafka, and app workloads into their own.

kubectl

The CLI you drive the cluster with — muscle memory for you already.

kind

"Kubernetes in Docker" — spins up a throwaway local cluster for testing before you touch cloud spend.

Amazon EKS

AWS's managed Kubernetes — the cloud target the book deploys the whole stack onto. Your daily driver.

eksctl

The CLI that provisions EKS clusters with one command; the book uses it to stand up the data platform.

Helm / Helm Chart

Kubernetes' package manager; nearly every data tool here (Spark Operator, Strimzi, Trino, Superset) installs via a chart.

The Modern Data Stack

03

Data Architecture & the Lakehouse

Data world Chapter 4

This is the mental model everything else hangs off. If you learn only one section deeply, make it this one — Lambda architecture and the bronze/silver/gold lakehouse are the blueprints the whole book implements on Kubernetes.

Big Data

Data too large, fast, or varied for a single machine or a classic database — the "volume, velocity, variety" problem this whole stack exists to solve.

Data Warehouse

The classic analytics store: structured data, schema defined upfront, loaded in batches. Reliable but rigid and pricey at scale.

Data Lake

A central store for raw data of any type on cheap object storage, schema applied only when read — flexible, but ungoverned it becomes a swamp.

Data Lakehouse

The best of both: a lake's cheap flexible storage plus a warehouse's SQL performance and reliability — the architecture this book targets.

Lambda Architecture

A pattern with parallel batch and speed layers merged by a serving layer — historical depth plus real-time freshness in one query surface.

Kappa Architecture

Lambda simplified to a single streaming path — everything, even history, flows through one stream (replayed from a log like Kafka). Simpler design, harder to operate.

Batch Layer

Processes the full dataset in bulk on a schedule to produce accurate, immutable views — optimised for throughput, powered by Spark.

Speed Layer

Handles just-arrived data in real time to fill the gap until the next batch run — optimised for low latency, powered by Kafka + streaming.

Serving Layer

Answers queries by merging batch and speed views so consumers see one unified, up-to-date picture — Trino's job here.

Medallion Architecture

Organising the lakehouse into bronze (raw), silver (cleaned), and gold (aggregated) tiers, each refining the last.

Schema-on-Read

Store data raw now, decide its structure only when you query it — the flexibility that makes data lakes work.

Schema-on-Write

The warehouse way: define structure before loading. Safer but slow to adapt to new sources.

ETL / ELT

Extract-Transform-Load moves and reshapes data between systems; ELT flips the order — load raw first, transform in place — the lakehouse style.

Structured / Unstructured Data

Structured = neat rows and columns; unstructured = text, images, video; semi-structured = JSON/XML in between. Lakes hold all three.

ACID

Atomic-Consistent-Isolated-Durable — the correctness guarantees that lakehouse formats bring to cheap storage so updates don't corrupt data.

Upsert

Insert-or-update in one operation; lakehouse table formats add this to object storage, which normally can't edit a single row.

Data Swamp

A data lake gone bad — undocumented, ungoverned, and unusable. The cautionary tale that motivates cataloging and the lakehouse.

Delta Lake

An open table format that layers ACID transactions, versioning, and schema enforcement on top of Parquet files in object storage.

Apache Hudi

A rival open table format with the same goal — transactional upserts and incremental reads on a data lake.

Apache Iceberg

Another leading open table format, known for large-scale schema evolution and hidden partitioning. Delta/Hudi/Iceberg are the "big three."

Object Storage

Amazon S3

Cheap, effectively infinite cloud storage that holds files as objects — the foundation every layer of the lake sits on.

Parquet

A compressed, columnar file format that's the default for analytics — reads only the columns a query needs, so it's fast and small.

Columnar Storage

Storing values column-by-column instead of row-by-row, so analytical queries scan far less data. Why Parquet beats CSV for analytics.

HDFS / Hadoop

The original big-data distributed filesystem and processing framework; largely superseded by cloud object storage + Spark, but the ancestor of it all.

Data Governance

Cataloging, access control, and quality rules that keep a lake discoverable and trustworthy rather than a swamp.

04

Apache Spark

Data world Chapter 5

The workhorse engine for processing data at scale. The good news: Spark's architecture — a coordinator handing work to distributed executors — will feel familiar, because it's the same shape as Kubernetes itself. The genuinely new ideas are lazy evaluation and the shuffle.

Apache Spark

The dominant open-source engine for distributed data processing — splits big datasets across a cluster and crunches them in parallel, in memory.

PySpark

The Python API for Spark — how you'll actually write Spark jobs, given you're learning Python alongside this.

Driver Program

The coordinator process that plans the work and hands tasks to executors — the "brain" of a Spark application.

SparkSession

Your entry point into Spark — the object you create first that gives access to DataFrames and SQL.

SparkContext

The lower-level connection to the cluster that the SparkSession wraps — the plumbing to the cluster manager.

Cluster Manager

Allocates compute across the cluster and launches executors — can be YARN, Mesos, or, in this book, Kubernetes.

↳ FOR YOU When Spark runs on K8s, the cluster manager is your scheduler. Resource requests, node selection, autoscaling — all things you already tune.

Executor

A worker process (a pod, on K8s) that runs tasks and holds data in memory for one Spark application. More executors = more parallelism.

Task

The smallest unit of work — one operation on one data partition, run in a single thread inside an executor.

Stage

A group of tasks that can run without shuffling; a shuffle marks the boundary where one stage ends and the next begins.

Job (Spark)

The full computation triggered by a single action, split into stages and then tasks for distributed execution.

Slot / Core

A unit of parallelism inside an executor — the number of tasks it can run at once equals its slots (cores).

DataFrame

A distributed table with named, typed columns — the main way you work with data in Spark. Think pandas, but spread across a cluster.

RDD

Resilient Distributed Dataset

Spark's original low-level distributed collection; DataFrames are built on top and are what you'll normally use.

Spark SQL

Write plain SQL against Spark DataFrames — the familiar on-ramp for querying big data without learning a new API.

Transformation

An operation that defines a new dataset from an existing one (filter, join) — but doesn't run yet. It's only recorded.

Action

An operation that actually triggers execution and returns a result (count, collect, write). Nothing computes until an action fires.

Lazy Evaluation

Spark stacks up transformations without running them, then optimises the whole chain when an action is called — fewer, smarter passes over the data.

Narrow Transformation

Work each partition can do alone, no data movement (filter, map) — cheap and fast.

Wide Transformation

Work that needs data regrouped across partitions (groupBy, join) — forces a shuffle, the expensive operation.

Shuffle

Redistributing data across executors over the network so related records sit together — the main thing that makes Spark jobs slow.

Partition (data)

A chunk of a dataset that one task processes; how many you have controls how parallel — and how balanced — a job is.

Adaptive Query Execution

AQE

Spark re-optimising a query at runtime using real data statistics — re-balancing partitions and shuffles on the fly.

JVM

Java Virtual Machine

The runtime Spark executes on; why installing Spark locally needs Java. PySpark talks to it under the hood.

Spark UI

Spark's built-in web dashboard (port 4040) showing jobs, stages, and where time goes — your first stop for debugging slow jobs.

Structured Streaming

Spark's module for processing live streams as a continuous series of tiny batches — the same DataFrame API, applied to Kafka feeds.

Micro-Batch

How Structured Streaming works: collect a few milliseconds of stream, process it as a mini batch, repeat — low latency with batch simplicity.

05

Apache Airflow

Data world Chapter 6

The orchestrator — it decides what runs when and in what order across your pipeline. If you've used any CI/CD scheduler, the DAG concept will land immediately. The KubernetesExecutor is your bridge: it runs each task as a pod.

Apache Airflow

The open-source tool for authoring, scheduling, and monitoring data pipelines as code — the conductor of the whole batch orchestra.

DAG

Directed Acyclic Graph

A pipeline expressed as a one-way graph of tasks with dependencies — the core object you build in Airflow. "Acyclic" = no loops.

Operator

A pre-built task template — one operator = one step. You chain them into a DAG. Airflow ships dozens for common jobs.

PythonOperator

Runs an arbitrary Python function as a task — the most flexible operator, and where your Python practice pays off.

BashOperator

Runs a shell command as a task — handy for glue steps and calling out to CLIs.

Sensor

A special operator that waits for a condition (a file lands, a time passes) before letting the pipeline proceed.

Task Instance

One specific run of one task for one scheduled date — the atom Airflow tracks state for (queued, running, success, failed).

XCom

"Cross-communication" — the mechanism for passing small bits of data between tasks in a DAG.

Scheduler (Airflow)

The brain that reads DAGs, checks dependencies, and decides which task instances to queue next — scalable to multiple copies.

Metadata Database

The Postgres DB that stores all DAG and task state — Airflow's single source of truth that every component reads and writes.

Web Server

Hosts the Airflow UI (and a REST API) where you watch, trigger, and debug pipelines.

Executor (Airflow)

The component that actually runs tasks. LocalExecutor (one machine, testing), CeleryExecutor (worker pool), KubernetesExecutor (a pod per task).

↳ FOR YOU Choosing and operating the KubernetesExecutor in production is squarely a platform-engineering job — your lane.

Worker

The process (or pod) that executes an individual task the executor hands it.

Backfilling

Running a DAG over past dates it missed or that you added later — reprocessing history on demand.

Orchestration

Coordinating many separate steps and tools into one reliable, ordered, monitored workflow — Airflow's whole purpose.

Astro CLI / Astronomer

A command-line tool from Astronomer that spins up a full local Airflow with one command — how the book installs it painlessly.

Race Condition

A concurrency bug where the outcome depends on unpredictable timing between processes — Airflow's DB locks exist to prevent it. Familiar SRE territory.

06

Apache Kafka

Data world Chapter 7

The real-time backbone — a durable, replayable log that decouples the systems producing data from those consuming it. The partition is the key idea: it's how Kafka scales and stays fault-tolerant, and the mechanics of leaders/followers/replication will feel like distributed-systems territory you know.

Apache Kafka

A distributed platform for streaming events — a durable, replayable log that decouples the systems producing data from those consuming it.

Event Streaming

Treating data as a continuous flow of events processed as they happen, rather than files loaded in bulk — the real-time paradigm.

Publish-Subscribe

The messaging model where producers publish to a topic and any number of consumers subscribe — senders and receivers never talk directly.

Message Broker

The middleman that stores and routes messages between producers and consumers — what Kafka fundamentally is.

Topic

A named stream of records — the category you publish to and subscribe from. The unit you organise data flows around.

Partition (Kafka)

A topic is split into partitions — ordered, append-only shards spread across brokers. This is how Kafka scales and parallelises.

Producer

A client that writes (publishes) records to a topic — your apps, or Kafka Connect pulling from databases.

Consumer

A client that reads records from a topic and processes them — e.g. a Spark Structured Streaming job.

Consumer Group

A set of consumers sharing the work of a topic, each taking different partitions — how you scale out reading horizontally.

Offset

A consumer's bookmark — the position in a partition it has read up to. Because Kafka keeps the log, you can rewind and replay.

Broker

A single Kafka server holding partitions and serving reads/writes; a cluster is several brokers sharing the load.

Leader & Follower

Per partition, one broker is the leader (takes all reads/writes) and others are followers copying it; if the leader dies, a follower takes over.

Replication Factor

How many copies of each partition exist across brokers — the dial for durability. Factor 3 = one leader + two followers.

In-Sync Replica

ISR

A follower fully caught up with its leader; producers can wait for all ISRs to acknowledge a write before considering it safe.

Zookeeper

The coordination service Kafka has traditionally used to track brokers and elect partition leaders — deployed alongside the cluster.

Confluent Kafka

An enterprise-ready Kafka distribution maintained by Confluent Inc. — the container images the book uses locally.

Kafka Connect

A framework for moving data in and out of Kafka via reusable connectors — no code needed to sync a database into a topic.

Connector (Source / Sink)

A Kafka Connect plugin: a source pulls data from an external system into Kafka, a sink pushes from Kafka out to one (DB, Elasticsearch).

JDBC

The standard Java way to connect to SQL databases; the JDBC connector is how the book streams a SQL DB into Kafka.

Retention

How long Kafka keeps records before deleting them (default 7 days) — Kafka is a buffer, not permanent storage; data is synced to the lake for history.

07

Running the Data Stack on Kubernetes

Bridge Chapter 8

This is your chapter. Deploying Spark, Kafka, and Airflow on K8s runs entirely on the operator pattern — and you already operate operators (Karpenter, Argo CD). Once you see that Spark-on-K8s and Kafka-on-K8s are just more CRDs and controllers, the whole book snaps into your existing expertise.

Operator Pattern

Packaging the human know-how to run a complex app as a Kubernetes controller — it watches custom resources and manages the app for you.

Custom Controller

The code at an operator's heart that runs the reconcile loop — driving actual state toward the state your YAML declared.

Custom Resource Definition

CRD

Teaches Kubernetes a brand-new object type (like SparkApplication) so you can manage data tools with plain kubectl apply.

Custom Resource

CR

An instance of a CRD — the actual YAML you write to ask for a Spark job or a Kafka cluster.

Reconcile Loop

The controller's endless "compare desired vs actual, fix the difference" cycle — the mechanism behind self-healing. Core K8s intuition you have.

Declarative Management

You state the end result you want; the operator figures out the steps — the K8s philosophy, now applied to data infra.

Spark Operator

The operator (a Kubeflow project) that adds the SparkApplication CRD so you can run Spark jobs as native Kubernetes objects.

SparkApplication

The custom resource the Spark Operator introduces — a YAML file fully describing a Spark job the operator then runs as driver + executor pods.

Strimzi

The CNCF operator for running Kafka on Kubernetes — brings CRDs for clusters, topics, users, and connectors so Kafka becomes declarative.

CNCF

Cloud Native Computing Foundation

The open-source foundation that stewards Kubernetes and many operators (Strimzi, Prometheus) — the neutral home of the cloud-native ecosystem.

JBOD

Just a Bunch of Disks

Attaching several independent PersistentVolumes to each Kafka broker for more storage and throughput — a Strimzi storage option.

Ephemeral Cluster

A Kafka deployment using temporary storage — quick to stand up for testing, but data vanishes if pods restart. The disposable counterpart to JBOD/persistent.

Kafka Connect on K8s

Running the Connect framework itself as a Strimzi-managed deployment, so your DB-to-topic connectors scale and heal like any other workload.

08

The Data Consumption Layer

Data world Chapter 9

The "last mile" — how analysts actually query the lake. Two paths: Trino for SQL directly on batch data in S3, and Elasticsearch + Kibana for real-time search and dashboards. Notably, ES and Kibana overlap heavily with the ELK stack you already run for logs.

Data Lake Engine

A query engine that runs SQL directly on files in the lake — no loading into a warehouse first. The modern alternative Trino embodies.

Trino

formerly PrestoSQL

A distributed SQL engine that queries data straight from object storage (and many other sources) at petabyte scale — the book's serving layer.

Coordinator Node

Trino's planner — parses your SQL, builds a distributed plan, and farms it out to workers. (Same coordinator/worker shape as Spark.)

Trino Worker

A node that executes part of a query in parallel and returns results to the coordinator — scale these out for more query power.

MPP

Massively Parallel Processing

Splitting one query across many machines that each work on a slice — the design that lets Trino stay interactive on huge data.

ANSI SQL

The standard SQL dialect Trino speaks — so existing SQL skills and BI tools connect with no relearning.

DBeaver

A free desktop SQL client the book connects to Trino — a familiar GUI for running queries against the lake.

Elasticsearch

A distributed search-and-analytics engine (built on Lucene) that indexes JSON documents for fast full-text search and real-time analytics.

↳ FOR YOU The same Elasticsearch from the ELK logging stack you already operate — here it's a data-consumption tool, not just a log store.

Apache Lucene

The underlying search library Elasticsearch is built on — the engine that makes text search fast.

Inverted Index

The data structure behind full-text search: a map from each word to the documents containing it — why Elasticsearch finds matches instantly.

Full-Text Search

Searching the actual words inside documents (not just exact field matches) — Elasticsearch's core strength.

Index (Elasticsearch)

A collection of related JSON documents — Elasticsearch's rough equivalent of a database table. You "index" data to make it searchable.

Kibana

Elasticsearch's visualization front-end — dashboards, charts, and time-series exploration that update live as data streams in.

Apache Superset

An open-source BI and dashboarding tool (born at Airbnb) that connects to Trino and other SQL sources — deploys on K8s via Helm.

Data Visualization

Turning query results into charts and dashboards so humans can spot trends — the payoff at the end of every pipeline.

09

Generative AI on Kubernetes

Data world Chapter 11

The book's closing build — a GenAI app on K8s using managed models from Amazon Bedrock. If you went through the first field guide, this is familiar ground; here it's lighter, using Bedrock so you don't manage GPUs or model serving yourself.

Amazon Bedrock

AWS's managed service for foundation models — call top models via an API without hosting any GPUs. The book's shortcut to production GenAI.

Foundation Model

A large model pre-trained on massive data that you adapt rather than train from scratch — the base you build GenAI apps on.

RAG

Retrieval-Augmented Generation

Feeding a model your own documents at query time so it answers from your data instead of guessing — grounds answers and cuts hallucination.

Knowledge Base

Bedrock's managed RAG store — point it at documents in S3 and it handles the embedding and retrieval so your app just asks questions.

Embedding

A numeric vector capturing a piece of text's meaning, so similar meanings sit close together — what makes semantic retrieval work in RAG.

Embeddings Model

The model that converts text or images into those vectors — the first step when loading documents into a knowledge base.

Agent

A model given tools and permission to act — it can call APIs and chain steps to complete a task, not just answer. Bedrock Agents automate workflows.

Streamlit

A Python library that turns a script into a simple web app fast — the book uses it to build the GenAI chat UI, then deploys it on K8s.

10

Production Concerns & Next Steps

Bridge Chapter 12

The book's final chapter is a tour of what to add for production — and it reads like your résumé. Monitoring, service mesh, security, autoscaling, GitOps, cost control: you can skim these as confirmation. For a data platform, the emphasis shifts to stateful scaling and cost, where data workloads bite hardest.

Monitoring

Watching cluster and app health via metrics, logs, and alerts — for a data platform, watch pipeline lag and job failures, not just pods.

Prometheus

The standard K8s metrics collector — you already run it; operators like Strimzi expose data-tool metrics straight into it.

Grafana

The dashboards-and-alerts layer over your metrics — your existing skill, now pointed at Spark, Kafka, and Airflow health.

Service Mesh

An infrastructure layer handling secure, observable service-to-service traffic (mTLS, retries, telemetry) without app code changes.

Istio

The most feature-rich service mesh — powerful traffic control and security, at the cost of complexity.

Linkerd

A lighter, simpler service mesh — the pragmatic alternative to Istio when you want mTLS and metrics without the overhead.

RBAC

Role-Based Access Control

Kubernetes' permission system — who can do what. Critical when data pipelines touch sensitive datasets. Familiar territory.

Network Policy

Firewall rules for pod-to-pod traffic — lock down which components can reach your brokers and databases.

Cluster Security

Defense-in-depth for the cluster and its data — RBAC, network policies, secrets management, image scanning. Your SRE bread and butter.

Autoscaling

Adding/removing capacity automatically with load — for data, it means scaling executors and consumers, and the nodes under them.

HPA

Horizontal Pod Autoscaler

Adds or removes pod replicas based on metrics like CPU — the pod-level scaler you already configure.

KEDA

Event-Driven Autoscaling

Scales workloads on external signals — like Kafka consumer lag, so you add consumers exactly when the queue backs up. Ideal for data.

Karpenter

Provisions right-sized nodes on demand — you already run this; for data, it means spinning up big nodes for a Spark job then releasing them.

GitOps

Git as the single source of truth for cluster state, auto-synced by a controller — your ArgoCD expertise applies directly to data infra.

Argo CD

The GitOps controller that continuously syncs your cluster to a Git repo — deploy the whole data stack by committing YAML. Your daily tool.

Flux

The other major GitOps engine — same commit-to-deploy model as Argo CD, different implementation.

CI / CD

Automated build-test-deploy pipelines — extended here to ship pipeline code and data infrastructure, not just apps.

Cost Control

Kubecost

Tracking and attributing K8s spend — acute for data, where idle GPU/large nodes and always-on brokers dominate the bill. Your FinOps angle.

PCI DSS

A security-compliance standard for handling payment-card data — the kind of governance data platforms must satisfy.
